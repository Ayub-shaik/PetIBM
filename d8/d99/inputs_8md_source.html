1 # Input files
2 
3 PetIBM reads input parameters from a YAML configuration file (typically called `config.yaml`) that is located in the simulation directory (i.e. directory where the numerical solution will be written).
4 
5 If the current working directory is not the simulation directory, the user should provide the command-line argument `-directory <simulation-directory>`.
6 
7 If the configuration file is located in another directory, the user should provide the path of the file using the command-line argument `-config <file-path>`.
8 (The path can be absolute or relative to the current working directory.)
9

12 ## Basic YAML configuration file
13 
14 A basic YAML configuration file for a 2D run will look like:
15 
16 ```yaml
17 mesh:
18  - direction: x
19  start: 0.0
20  subDomains:
21  - end: 1.0
22  cells: 32
23  stretchRatio: 1.0
24  - direction: y
25  start: 0.0
26  subDomains:
27  - end: 1.0
28  cells: 32
29  stretchRatio: 1.0
30 
31 flow:
32  nu: 0.01
33  initialVelocity: [0.0, 0.0]
34  boundaryConditions:
35  - location: xMinus
36  u: [DIRICHLET, 0.0]
37  v: [DIRICHLET, 0.0]
38  - location: xPlus
39  u: [DIRICHLET, 0.0]
40  v: [DIRICHLET, 0.0]
41  - location: yMinus
42  u: [DIRICHLET, 0.0]
43  v: [DIRICHLET, 0.0]
44  - location: yPlus
45  u: [DIRICHLET, 1.0]
46  v: [DIRICHLET, 0.0]
47 
48 parameters:
49  dt: 0.01
50  startStep: 0
51  nt: 1000
52  nsave: 1000
53  nrestart: 1000
54  convection: ADAMS_BASHFORTH_2
55  diffusion: CRANK_NICOLSON
56  velocitySolver:
57  type: CPU
58  config: solversPetscOptions.info
59  poissonSolver:
60  type: CPU
61  config: solversPetscOptions.info
62 
63 bodies:
64  - type: points
65  file: circle.body
66 ```
67 
68 The configuration contains three mandatory nodes: `mesh`, `flow`, `parameters`.
69 The node `bodies` is required when running a simulation with at least one boundary immersed in the computational domain.

73 ## YAML node `mesh`
74 
75 `mesh` controls the parameters of the structured Cartesian grid to be created.
76 We need to provide sufficient information to generate the gridline coordinates in the `direction` `x`, `y`, and `z`.
77 (For 2D runs, the `z` direction should be omitted.)
78 
79 A gridline is defined by the location of a starting point (`start`), which corresponds to the minimum value along the gridline (for example, location of the left boundary in the x-direction).
80 
81 A gridline is divided into consecutive sub-domains (`subDomains`), each one being defined with the coordinate of a ending point (`end`), a number of cells (`cells`), and optionally a stretching ratio (`stretchRatio`).
82 The stretching ratio represents the ratio between the width of two consecutive cells.
83 (A ratio greater than `1.0` means that the cell width increases along the gridline in the positive direction.)
84 
85 Based on those information, PetIBM computes the cell widths and the user should make sure that the cells at the interface between two sub-domains are approximately of equal size.
86 
87 Example:
88 
89 The following node will generate a 2D structured Cartesian grid with 1704 cells in the x and y directions in the domain [-15, 15]x[-15, 15] where the mesh is uniform in the sub-domain [-0.52, 1.48]x[-2, 2] and stretched to the external boundaries with a constant ratio of 1.01.
90 
91 ```yaml
92 mesh:
93  - direction: x
94  start: -15.0
95  subDomains:
96  - cells: 363
97  end: -0.52
98  stretchRatio: 0.9900990099009901
99  - cells: 1000
100  end: 3.48
101  stretchRatio: 1.0
102  - cells: 341
103  end: 15.0
104  stretchRatio: 1.01
105  - direction: y
106  start: -15.0
107  subDomains:
108  - cells: 352
109  end: -2.0
110  stretchRatio: 0.9900990099009901
111  - cells: 1000
112  end: 2.0
113  stretchRatio: 1.0
114  - cells: 352
115  end: 15.0
116  stretchRatio: 1.01
117 ```

121 ## YAML node `flow`
122 
123 `flow` prescribed the characteristics of the fluid as well as the initial and boundary conditions of the flow velocity.
124 
125 `nu` is the kinematic viscosity of the fluid.
126 
127 `initialVelocity` describes the initial uniform velocity vector field.
128 
129 `boundaryConditions` lists the type and value of a velocity component (`u`, `v`, or `w`) for all boundaries.
130 The boundary locations are: `xMinus` and `xPlus` for the left and right, `yMinus` and `yPlus` for the bottom and top, and `zMinus` and `zPlus` for the front and back boundaries.
131 (`zMinus` and `zPLus` should be omitted for 2D runs.)
132 PetIBM implements different type of boundary conditions:
133 
134 - Dirichlet (`DIRICHLET`); the value corresponds the value of the velocity component at the boundary;
135 - Neumman (`NEUMANN`); the value represents the value of the derivative normal to the boundary of the velocity component;
136 - convective (`CONVECTIVE`); the value is the speed at which the velocity component is convected;
137 - periodic (`PERIODIC`); the value has no meaning and can be set to whatever number.
138 
139 The following node corresponds to flow characteristics for a 3D cavity flow (initially at rest) at Reynolds number 500 (based on the kinematic viscosity, the length of the cavity, and the speed of the lid-driven wall) with a lid-driven wall at the top boundary (moving with speed 1 in the x direction) and with periodic boundary conditions in the z direction.
140 
141 ```yaml
142 flow:
143  nu: 0.002
144  initialVelocity: [0.0, 0.0, 0.0]
145  boundaryConditions:
146  - location: xMinus
147  u: [DIRICHLET, 0.0]
148  v: [DIRICHLET, 0.0]
149  w: [DIRICHLET, 0.0]
150  - location: xPlus
151  u: [DIRICHLET, 0.0]
152  v: [DIRICHLET, 0.0]
153  w: [DIRICHLET, 0.0]
154  - location: yMinus
155  u: [DIRICHLET, 0.0]
156  v: [DIRICHLET, 0.0]
157  w: [DIRICHLET, 0.0]
158  - location: yPlus
159  u: [DIRICHLET, 1.0]
160  v: [DIRICHLET, 0.0]
161  w: [DIRICHLET, 0.0]
162  - location: zMinus
163  u: [PERIODIC, 0.0]
164  v: [PERIODIC, 0.0]
165  w: [PERIODIC, 0.0]
166  - location: zPlus
167  u: [PERIODIC, 0.0]
168  v: [PERIODIC, 0.0]
169  w: [PERIODIC, 0.0]
170 ```

174 ## YAML node `parameters`
175 
176 This node gather various `parameters` regarding the advancement of a simulation.
177 
178 - `dt`: the time-step size.
179 - `startStep`: index of the stating time step (default is zero).
180 - `nt`: number of time steps to compute.
181 - `nsave`: frequency (in number of time steps) of saving for the numerical solution.
182 - `nrestart`: frequency (in number of time steps) of saving for the convective and diffusive terms; those terms will required upon restart of a run at a time step different from 0.
183 - `convection`: time scheme for the convective terms; choices are the default explicit Euler method (`EULER_EXPLICIT`) or an explicit second-order Adams-Bashforth scheme (`ADAMS_BASHFORTH_2`).
184 - `diffusion`: time scheme for the diffusive terms; choices are the default implicit Euler method (`EULER_IMPLICIT`), an explicit Euler method (`EULER_EXPLICIT`), or a second-order Crank-Nicolson scheme (`CRANK_NICOLSON`).
185 - `velocitySolver`, `poissonSolver`, and `forcesSolver` (for the decoupled version of the immersed-boundary projection method) each references the type of hardware used to solve the linear system (either `CPU` or `GPU`) and the path (relative to the YAML configuration file) of the file containing the parameters for the linear solver.
186 
187 In the following example, PetIBM will run 1000 time steps (from time step 0) with a time increment of 0.01, saving the numerical solution (velocity vector field, pressure scalar field, and Lagrangian boundary forces) every 100 time steps and saving the convective and diffusive terms every 200 time steps.
188 
189 ```yaml
190 parameters:
191  dt: 0.01
192  startStep: 0
193  nt: 1000
194  nsave: 100
195  nrestart: 200
196  convection: ADAMS_BASHFORTH_2
197  diffusion: CRANK_NICOLSON
198  velocitySolver:
199  type: CPU
200  config: solversPetscOptions.info
201  poissonSolver:
202  type: GPU
203  config: solversAmgXOptions.info
204 ```
205 
206 In the example above, the velocity system will be solved on CPU with PETSc KSP and the parameters of the linear solver can be prescribed in the file `solversPetscOptions.info`.
207 (The PETSc prefix for the arguments is `-velocity_`, `-poisson_`, and `-forces_` for the velocity system, the Poisson system, and the Lagrangian forces system, respectively.)
208 
209 The Poisson system will be solved on GPU devices with the [NVIDIA AmgX library](https://github.com/NVIDIA/AMGX) and the parameters of the linear solver are prescribed in the file `solversAmgXOptions.info`.

213 ## YAML node `bodies`
214 
215 `bodies` provides information about the boundaries immersed in the computational domain.
216 
217 For example, the following node stipulates that there are two immersed boundaries defined by `points` coordinates that can be read in the ASCII files `boundary1.txt` and `boundary2.txt`, respectively.
218 (The path of the files can be either absolute or relative to the simulation directory.)
219 
220 ```yaml
221 bodies:
222  - type: points
223  file: boundary1.txt
224  - type: points
225  file: boundary2.txt
226 ```
227 
228 The file `boundary1.txt` contains the coordinates of the first boundary with the first line storing the number of points.
229 The other lines contain the coordinates of points describing the boundary (2 columns for 2D simulations, 3 columns for 3D simulations).
230 
231 Example of a 2D boundary with 4 points:
232 ```
233 4
234 0.0 0.0
235 1.0 0.0
236 1.0 1.0
237 0.0 1.0
238 ```
