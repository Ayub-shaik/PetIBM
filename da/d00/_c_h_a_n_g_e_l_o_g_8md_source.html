65 ## 0.4.0
66 
67 ---
68 
69 ### Added
70 
71 * Add possibility to use configuration arguments `--with-<package>-include=<path>` and `--with-<package>-lib=<path>` (instead of `--with-<package>-dir=<path>`) for yaml-cpp, gtest, AmgX, and AmgXWrapper. (`--with-<package>-dir=<path>` is still supported for those packages.)
72 * Add classes to monitor the solution in sub-regions or at specific points (using linear interpolation). YAML configuration should be provided in the node "probes". (See Markdown documentation `doc/markdowns/inputs.md` for details on how to use.)
73 * Install application header files upon make install call. The header files are installed in the include folder of the install directory. A user can now create new classes that inherits from an application class.
74 * Add a simple class `RigidKinematicsSolver` in the applications folder to handle cases with moving rigid bodies (with prescribed kinematics). The user should create a class that inherits from `RigidKinematcsSolver` and that implements the methods to update the coordinates and velocity of the Lagrangian points, `setCoordinatesBodies` and `setVelocityBodies`.) For example, if you want to compute the flow around a flapping wing, you just have to create a class (outside the PetIBM source directory) that inherits from the application class `RigidKinematicsSolver` and implement the case-specific methods to update the location and velocity of the wing.
75 * Add possibility to implement different kernels for the regularized delta function. (The 3-point delta function from Roma et al. (1999) and the 4-point delta function from Peskin (2002) are currently available in PetIBM.)
76 
77 ### Changed
78 
79 * Upgrade to yaml-cpp-0.6.2 when downloading building yaml-cpp at PetIBM configuration time (configuration flag `--enable-yamlcpp`).
80 * Re-format code with clang-format. The clang-format style file is added to the repository.
81 * Re-write I/O functions to provide file paths that include the extension.
82 * Re-write application codes to simplify main functions (move PetIBM object instantiations into the init method of the application class).
83 * Change the scheme for the prediction of the forces at the beginning of a time step. Now using the forces from the previous time steps as prediction. (This is named "scheme 2" in Li et al., 2016.) Using this scheme avoids having to reset the vector of the Lagrangian momentum forcing every time step. (According to the authors there is little difference in the results between the different forcing schemes.)
84 * For each example, move the configuration files for the solvers into the sub-folder `config` of the simulation directory.
85 * Update the example for the 2D in-line oscillating cylinder to use the newly implemented application class `RigidKinematicsSolver`.

123 ## 0.3.0
124 
125 ---
126 
127 WARNING: this is a major re-factorization of the code that is not backward compatible.
128 
129 ### Added
130 
131 * PetIBM is now a library.
132 * Add `include` folder with all header files.
133 * Add `applications` folder with code applications that use the PetIBM library.
134 * Add some unit-tests.
135 * Add instructions on how to contribute (`CONTRIBUTING.md`).
136 * Add multi-body example (2 vertically aligned cylinders at Re=100).
137 * Add moving-body example (oscillating cylinder at Re=100).
138 * Require PETSc-3.8.
139 
140 ### Changed
141 
142 * Code is now under the BSD 3-clause license.
143 * Move from a single PetIBM executable to a PetIBM library with a set of application codes (Navier-Stokes solver, immersed-boundary method solvers, and post-processing utilities to compute the vorticity field and generate XMDF files).
144 * Update user's documentation and API documentation. (Markdown files for user's documentation are now in `doc/markdowns`; API documentation is generated with Doxygen into `doc/hmtl`.)
145 * External dependencies such as yaml-cpp, Boost, Gtest, and AmgXWrapper are now optional; they can be installed separately or when configuring PetIBM.
146 * Application codes solve the systems for the velocity components, not anymore for the velocity fluxes.
147 * Update previous examples with new configuration file.
148 * Update Travis CI build to cache PETSc-3.8.2.
149 
150 ### Fixed
151 
152 
153 ### Removed
154 
155 * Remove all Python scripts from the `scripts` folder. (It contained old Python scripts that were not used anymore.)
156 * Cannot output the fields into PETSc binary format; only HDF5 is supported.
157 * Remove Wiki pages. (User's documentation and API documentation now available [here](https://barbagroup.github.io/PetIBM).)

161 ## 0.2.0
162 
163 ---
164 
165 ### Added
166 
167 * Compatibility with PETSc-3.7 (last tested 3.7.4; cannot use 3.5 and 3.6 anymore).
168 * A change log.
169 * Possibility to output the numerical solution in HDF5 format.
170 * Possibility to output the flux and/or velocity variables.
171 * Python script `createXMFFile.py` that generates a `.xmf` file to let VisIt know how to read the HDF5 files.
172 * [AmgXWrapper](https://github.com/barbagroup/AmgXWrapper) as an optional external package (version v1.0-beta).
173 * Examples to solve the 2D flow around a circular cylinder using AmgX.
174 * Possibility to have multiple immersed boundaries.
175 * Decoupled IBPM solver (as published in [Li et al., 2016](http://www.sciencedirect.com/science/article/pii/S0045793016302833))
176 
177 ### Changed
178 
179 * Comply Python pre- and post-processing scripts to PEP8.
180 * Move `validation_data` files to `scripts/python/verification/data` and `scripts/python/validation/data`.
181 * GAMG parameters for the 2D lid-driven cavity flow to make it run.
182 * Update external package AmgXWrapper to version 1.0-beta2.
183 
184 ### Fixed
185 * Update boundary ghost points only once per time step, at the beginning of the time step.
186 * Bug-fix in the methods `generateBNQ()` of the class `TairaColoniusSolver` and `generateET` of the class `LiEtAlSolver`: calculate the correct widths of the computational domain (fixed index).
187 
188 ### Removed
189 
190 * Python script `restartFromSolution.py` (not finished and not necessary).
191 * Non-zero initial guess as default for KSPs; the user should add `-ksp_initial_guess_nonzero true` to the command-line (or configuration file) to switch on a nonzero initial for a specific KSP.
